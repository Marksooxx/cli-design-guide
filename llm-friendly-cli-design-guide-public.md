# 面向 LLM 和 Agent 的 CLI 设计指南

命令行工具早已不只是给人类终端用户使用。越来越多的 CLI 会被 LLM、自动化脚本、IDE agent、MCP 适配层和 CI 流程调用。对这些调用方来说，CLI 不只是一个“能跑”的入口，更是一种需要被探索、理解、纠错和稳定消费的接口。

一套真正对 agent 友好的 CLI，不是简单地“支持 JSON 输出”就够了。它需要同时具备以下能力：

- 可发现：模型能通过 `--help` 和示例快速学会如何调用
- 可预测：相同输入在不同上下文中产生稳定、可预期的行为
- 可恢复：出错时能够自我纠正，危险操作有明确的安全边界
- 可解析：输出结构清晰，便于程序稳定消费
- 可契约化：字段、事件、错误和版本有明确约定，不会在升级后静默破坏集成方

本文给出一套面向公开发布的实践指南，目标不是把所有 CLI 设计成同一种样子，而是帮助你构建一套既适合人类使用、又适合 agent 自动调用的命令行接口。

---

## 1. 如何使用这份指南

本文中的建议分为三类：

- 基础要求：不满足会显著降低 agent 可用性
- 推荐实践：绝大多数 CLI 都值得采用
- 进阶能力：适合成熟工具或特定场景

如果你正在设计一个新 CLI，先满足基础要求，再逐步补齐推荐实践；如果你在改造现有工具，优先处理输出契约、错误模型和非交互执行这三件事。

---

## 2. 先设计信息架构，再设计命令词序

CLI 的命令词序通常有两种主流风格：

- 资源中心型：`noun verb`
  例：`mytool project create`、`mytool config get`
- 任务中心型：`verb noun` 或顶层动词
  例：`docker run`、`git commit`、`kubectl get pods`

生态里并不存在一种放之四海而皆准的“正确词序”。决定词序的关键，不是模仿某个明星工具，而是先回答两个问题：

- 用户和 agent 是先想到“对什么对象做什么”，还是先想到“要做什么动作”
- 你的工具是围绕资源组织，还是围绕任务组织

推荐实践：

- 在同一 CLI 内保持风格一致，不要一半是资源中心、一半是任务中心
- 命令层级尽量控制在 3 层以内
- 高频操作可以保留顶层快捷命令，例如 `init`、`run`

层级过深会直接增加 agent 的记忆负担。相比堆叠更多中间名词，更好的做法通常是把深层概念下沉为 flag、配置项或输入参数。

---

## 3. Flag 设计首先追求惯例一致

对 agent 来说，flag 命名是否符合社区惯例，往往比文档是否优美更重要。名字越符合预期，模型越不需要反复查 `--help`。

推荐实践：

- 同时提供短 flag 和长 flag
- 在文档和集成说明中明确建议 agent 使用长 flag
- 为布尔选项提供成对形式，例如 `--cache` / `--no-cache`

常见命名约定如下：

| 用途 | 优先命名 |
|---|---|
| 输出路径 | `--output` |
| 输入路径 | `--input` 或位置参数 |
| 强制执行 | `--force` |
| 跳过确认 | `--yes` / `-y` |
| 静默模式 | `--quiet` / `-q` |
| 详细输出 | `--verbose` / `-v` |
| 干跑 | `--dry-run` |
| 配置文件 | `--config PATH` |
| 非交互模式 | `--non-interactive` 或 `--no-input` |

关于输出格式：

- 如果工具存在多种输出格式，优先使用 `--format json|yaml|text`
- 如果工具只有“默认文本 + JSON”两种输出，单独提供 `--json` 也可以接受
- 无论采用哪种形式，都应保持全工具一致

关于 `--verbose` 和 `--debug`：

- `--verbose` 适合表示更详细的正常运行信息
- `--debug` 可以存在，但它应表示额外的开发诊断信息，而不是与 `--verbose` 混用

---

## 4. 输出设计的核心不是 JSON，而是契约

支持 JSON 输出只是第一步。真正决定 agent 能否稳定集成的，是输出是否有清晰且长期稳定的契约。

### 4.1 结构化输出应显式开启

基础要求：

- 提供结构化输出能力
- 让调用方能够显式选择，例如 `--format json`
- 不要要求调用方依赖 TTY 状态猜测当前输出格式

示例：

```bash
$ mytool list
NAME       STATUS    SIZE
audio.wav  ok        2.3MB
voice.ogg  pending   1.1MB

$ mytool list --format json
{
  "schema_version": "1",
  "status": "ok",
  "data": [
    {"name": "audio.wav", "status": "ok", "size_bytes": 2411724},
    {"name": "voice.ogg", "status": "pending", "size_bytes": 1153434}
  ],
  "errors": [],
  "warnings": [],
  "meta": {}
}
```

显式传参的好处是输出行为稳定。无论在终端、CI、管道、重定向还是 agent 调用环境里，同一条命令都不会因为运行上下文不同而悄悄改变输出格式。

### 4.2 默认输出不要随 TTY 状态漂移

推荐实践：

- TTY 检测只用于控制颜色、进度条、交互提示等展示层特性
- 不要用 TTY 检测在“表格”和“JSON”之间自动切换

如果确实希望提供自动格式选择，应让它成为显式 opt-in 能力，例如 `--auto-format`，而不是默认行为。

### 4.3 stdout 和 stderr 必须分工明确

基础要求：

- `stdout` 输出主结果
- `stderr` 输出日志、进度、警告和辅助诊断信息

在结构化模式下，推荐把 `stdout` 视为唯一的机器消费通道。这意味着：

- 成功结果写入 `stdout`
- 结构化错误结果也写入 `stdout`
- 任何不属于结果契约的日志和进度都写入 `stderr`

这样做的好处是，调用方始终只需解析一个稳定通道，而不会在日志和结果之间反复分流。

### 4.4 为结构化输出设计稳定的结果包

推荐实践：

- 在顶层包含 `schema_version`
- 顶层字段保持稳定
- 新字段优先追加，不要直接重命名或删除旧字段
- 列表为空时返回空数组，而不是省略字段
- 所有自动化流程都依赖的关键字段，必须在文档中明确声明

一套简单而稳定的结果包通常就足够：

```json
{
  "schema_version": "1",
  "status": "ok",
  "data": {},
  "errors": [],
  "warnings": [],
  "meta": {}
}
```

其中：

- `status` 建议使用 `ok`、`partial`、`error`
- `data` 承载主结果
- `errors` 承载结构化错误项
- `warnings` 承载不会阻断执行、但值得上游处理的告警
- `meta` 承载游标、分页、耗时、请求 ID 等附加信息

### 4.5 版本化要早做，不要等破坏兼容后补救

推荐实践：

- 在所有结构化输出中包含 `schema_version`
- 约定兼容策略，例如“新增字段允许，删除字段需要弃用周期”
- 重大变更通过新版本 schema 或新格式选项显式启用

对于需要长期被脚本和 agent 调用的 CLI，这一点比是否支持 YAML 更重要。

---

## 5. 错误模型决定 agent 是否会自救

很多 CLI 最大的问题不是失败，而是失败得不够可操作。对人类来说，“invalid input”已经令人烦躁；对 agent 来说，这几乎等于没有信息。

### 5.1 文本错误至少要包含三部分

基础要求：

- 问题是什么
- 原因是什么
- 下一步该怎么改

示例：

```text
Error: --format value 'mp4' is not supported
Reason: 'audio convert' only accepts audio formats
Hint: try one of: wav, ogg, flac, mp3
See: mytool audio convert --help
```

这类错误对人类清晰，对 agent 也足够友好，因为模型可以直接根据 `Hint` 进行自我修正。

### 5.2 结构化错误应该和结果包兼容

基础要求：

- 在结构化模式下提供机器可读的错误对象
- 错误对象至少包含 `code`、`message`、`hint`

示例：

```json
{
  "schema_version": "1",
  "status": "error",
  "data": null,
  "errors": [
    {
      "code": "INVALID_FORMAT",
      "message": "format 'mp4' not supported",
      "hint": "use one of: wav, ogg, flac, mp3",
      "param": "--format"
    }
  ],
  "warnings": [],
  "meta": {}
}
```

相比只输出一段文本，稳定的错误码能显著降低上层集成逻辑的脆弱性。

### 5.3 拼写纠错是高收益的小功能

推荐实践：

- 对未知子命令和未知 flag 提供“Did you mean”
- 推荐候选不要过多，通常 1 到 3 个就够

示例：

```text
Error: unknown subcommand 'conver'
Did you mean: convert?
```

### 5.4 退出码应该简单、稳定、文档化

推荐实践：

- `0`：完整成功
- `1`：执行失败或部分失败，具体细节看结构化结果
- `2`：调用方式错误，例如参数错、子命令不存在、必填参数缺失
- `130`：被中断

如果工具需要更细粒度的退出码，可以继续扩展，但应尽量保持 `1` 和 `2` 的高层语义稳定。对 agent 来说，“调用错了”与“执行时出错了”是最关键的分界线。

## 6. 安全边界要真实存在，而不是看起来存在

对 agent 来说，最危险的 CLI 不是“功能强”，而是“默认破坏性强、预览又不可靠”。

### 6.1 破坏性命令默认应受保护

基础要求：

- 删除、覆盖、外发、部署、写入远端等操作，默认必须有保护措施
- 保护措施可以是确认、显式 `--yes`、双阶段执行，或更安全的语义设计

例如：

```text
mytool delete file.wav
```

不应直接执行不可逆删除。更稳妥的设计包括：

- 默认移动到回收站，`--hard` 才是真删除
- 默认失败并提示需要 `--yes`
- 先生成计划，再显式执行

### 6.2 dry-run 只在语义成立时提供

推荐实践：

- 对本地文件写入、批量重命名、配置改写等操作提供 `--dry-run`
- 对部署类任务优先提供 `--plan`
- 对远程副作用操作，如果无法保证 dry-run 语义真实成立，就明确说明不支持

错误做法不是“没有 `--dry-run`”，而是提供一个看似安全、实际不等价的假 dry-run。对 agent 来说，这比没有预览能力更危险。

### 6.3 对可重试的副作用操作，显式支持幂等

推荐实践：

- 文档中明确哪些命令天然幂等，例如 `get`、`list`、`describe`
- 对外发、创建、扣费、通知等副作用命令支持 `idempotency key`

例如：

```bash
mytool send-email --idempotency-key 6d8b7c4a-...
```

agent 一旦因为超时或网络抖动重试，没有幂等保障就很容易把“安全重试”变成“重复执行”。

### 6.4 为真实世界的问题保留审计能力

推荐实践：

- 提供操作历史或操作详情查询
- 在结果包中返回操作 ID、请求 ID 或任务 ID

`undo` 可以是很好的进阶能力，但它只适用于语义上真正可逆的操作，不应被包装成通用承诺。

---

## 7. 非交互模式必须是一等公民

agent 没有可靠的 stdin，很多自动化环境也不允许交互输入。一个只适合“人在终端里手动按回车”的 CLI，很难成为合格的自动化接口。

基础要求：

- 提供 `--yes`、`--non-interactive` 或等价能力
- 非交互环境遇到本应确认的步骤时，默认应失败并清晰报错
- 不要在非交互模式下静默选择默认值

错误示例：

- 进入非 TTY 环境后自动选“yes”
- 没有任何提示地跳过警告
- 突然弹出密码输入，导致进程挂死

推荐实践：

- 凭证优先支持环境变量
- 支持从文件或标准输入以外的安全渠道读取凭证
- 在错误信息中明确指出需要加什么参数才能在非交互模式下继续执行

---

## 8. 长任务、流式输出和批量语义要单独设计

很多 CLI 的输出设计只考虑了“命令执行完之后打印一段结果”，但真实世界里常见的还有：

- 长时间运行的处理任务
- 批量操作
- watch 模式
- 部分成功、部分失败的执行结果

这些场景如果还强行套用一次性文本输出，agent 体验会迅速恶化。

### 8.1 长任务优先提供 JSON Lines 或 NDJSON

推荐实践：

- 快照类查询使用普通 JSON
- 流式任务使用 JSON Lines 或 NDJSON
- 每条事件都包含稳定的 `event` 字段

示例：

```text
{"event":"start","total":100}
{"event":"progress","done":10,"current":"file_010.wav"}
{"event":"item","id":"file_010.wav","status":"ok"}
{"event":"item","id":"file_011.wav","status":"error","error":{"code":"TIMEOUT","message":"..."}}
{"event":"done","summary":{"total":100,"ok":97,"failed":3}}
```

这样 agent 能够增量消费结果，而不必等待一个巨大数组在最后一次性返回。

### 8.2 批量操作必须显式表达部分失败

基础要求：

- 批量任务不能只给成功项
- 必须能区分全部成功、全部失败和部分成功
- 退出码和结果包都应体现失败事实

推荐的结果包形态：

```json
{
  "schema_version": "1",
  "status": "partial",
  "data": {
    "summary": {"total": 100, "ok": 97, "failed": 3},
    "results": [
      {"id": "file_001", "status": "ok"},
      {
        "id": "file_042",
        "status": "error",
        "error": {
          "code": "TIMEOUT",
          "message": "upload timed out"
        }
      }
    ]
  },
  "errors": [],
  "warnings": [],
  "meta": {}
}
```

如果批量任务发生了部分失败，退出码至少不应表现为“完整成功”。

### 8.3 长任务应支持中断与恢复

推荐实践：

- 被中断时优雅退出，清理临时状态
- 可恢复任务支持 `--resume`
- 在输出中暴露 checkpoint、任务 ID 或恢复所需上下文

## 9. `--help` 是接口的一部分，不是附赠说明

对于 agent 来说，`--help` 往往是第一次接触你的 CLI。帮助信息写得好不好，会直接影响第一次调用是否成功。

基础要求：

- 每个子命令都有帮助文本
- 帮助文本至少包含用途、参数说明、示例、退出码说明
- 示例覆盖最常见的 2 到 3 个用法

一个可用的帮助页面通常应包含：

```text
USAGE:
    mytool audio convert <INPUT> [OPTIONS]

ARGUMENTS:
    <INPUT>    Input file or directory

OPTIONS:
    -o, --output <PATH>     Output path
    -f, --format <FORMAT>   Target format
        --dry-run           Preview without writing
        --yes               Skip confirmation prompts

EXAMPLES:
    mytool audio convert in.wav -f ogg
    mytool audio convert ./input -f ogg --dry-run

EXIT CODES:
    0  success
    1  execution failed or partially failed
    2  invalid invocation
```

推荐实践：

- 为命令树提供 `commands` 或等价的可枚举接口
- 提供 `config show`
- 为复杂工具提供 `doctor`
- 为成熟工具提供 `schema <cmd>` 输出命令结构定义

这些都不一定是基础门槛，但它们能显著提升 agent 的探索效率。

---

## 10. 配置与状态管理要对自动化友好

一个依赖隐式本地状态的 CLI，很难成为可靠的 agent 工具。

推荐实践：

- 明确配置优先级：`CLI flag > 环境变量 > 项目配置 > 用户配置 > 默认值`
- 为环境变量使用统一前缀，例如 `MYTOOL_`
- 支持项目级配置覆盖用户级配置

配置位置应遵循平台主流约定，并在文档中说清楚：

| 平台 | 推荐位置 |
|---|---|
| Linux | `~/.config/mytool/config.toml` |
| macOS | `~/Library/Application Support/mytool/` 或一致的 XDG 风格目录 |
| Windows | `%APPDATA%\mytool\config.toml` |
| 项目级 | `./mytool.toml` |

更重要的是保持一致，而不是执着于唯一正确路径。

基础要求之外，一个重要原则是：**尽量减少对隐藏状态的依赖**。能通过显式参数表达的上下文，不要只藏在某个本地缓存、最近一次运行结果或隐式会话里。

---

## 11. 启动性能会在 agent 循环里被放大

单次启动慢一点，人类也许能接受；但在 agent 循环里，几十次调用叠加后，体验会迅速变差。

推荐实践：

- 让 `--help` 和元信息命令尽量快
- 避免在解析参数之前加载重型依赖
- 使用 lazy import 或分层初始化

经验上：

- `--help` 和 `--version` 应明显快于业务命令
- 如果某类任务需要长时间初始化，可以考虑 `warmup` 或后台守护进程模式

不要让一个只想查看帮助信息的调用，顺带启动模型、连接数据库或扫描整个项目目录。

---

## 12. 为 agent 写集成说明时，尽量少而精

当 CLI 本身足够自描述时，配套的 agent 集成说明不需要很厚。无论它最终落在 `SKILL.md`、tool metadata、system prompt 还是 MCP 描述里，重点通常只有几条：

- 先从 `--help` 开始探索
- 优先使用长 flag
- 需要解析的输出始终显式请求结构化格式
- 非交互环境必须传 `--yes` 或 `--non-interactive`
- 对本地破坏性操作先用 `--dry-run`，对远程副作用操作关注幂等和 staging

一个好的 CLI，应该让集成说明更像“调用约定”，而不是“补齐 CLI 自身缺陷的外部文档”。

---

## 13. 设计 Checklist

### 基础要求

- [ ] 命令命名风格在工具内一致
- [ ] 支持显式结构化输出
- [ ] 默认输出格式不因 TTY 变化而漂移
- [ ] `stdout` 与 `stderr` 分工明确
- [ ] 错误信息包含问题、原因、建议
- [ ] 结构化错误至少包含 `code`、`message`、`hint`
- [ ] 退出码至少稳定区分调用错误和执行错误
- [ ] 破坏性命令默认受保护
- [ ] 支持非交互执行
- [ ] `--help` 含示例
- [ ] 批量操作能表达部分失败

### 推荐实践

- [ ] 长短 flag 双轨制
- [ ] 长 flag 语义清晰并遵循社区惯例
- [ ] 结构化输出包含 `schema_version`
- [ ] 长任务支持 JSON Lines 或 NDJSON
- [ ] 提供拼写纠错
- [ ] 命令层级尽量不超过 3 层
- [ ] 配置优先级清晰
- [ ] 凭证支持环境变量
- [ ] 副作用命令支持幂等重试机制

## 14. 常见反模式

以下做法会让 CLI 在 agent 场景下变得脆弱、危险或难以集成：

- 输出格式根据 TTY 状态自动变化
- 把日志、进度和结果混在同一个通道里
- 错误信息只有一行 `Error`
- 非交互环境里静默选择默认值
- 默认行为就是破坏性操作
- 提供语义不成立的假 `--dry-run`
- 批量任务只输出成功项，不表达失败项
- 结构化输出没有版本字段
- 输出字段经常漂移或重命名，却没有兼容策略
- 启动阶段加载大量与当前命令无关的依赖
- 强依赖隐式本地状态，导致相同命令在不同机器上行为不一致

## 15. 结语

最好的 LLM-friendly CLI，不是“看起来像 API”的 CLI，而是“本身就具备接口质量”的 CLI。

它应该做到三件事：

- 看一眼 `--help` 就能开始正确调用
- 看一眼错误信息就知道下一步怎么改
- 看一眼结构化输出就能稳定接入自动化系统

当这些能力真正落到 CLI 本体上时，面向 agent 的集成文档会自然变薄，自动化链路也会明显更稳。这时，CLI 就不再只是一个终端工具，而是一种对人类和机器都友好的接口。
